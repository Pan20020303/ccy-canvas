# 模型适配与节点参数条设计

- 日期：2026-05-28
- 适用范围：`/app` 工作区生成节点、`/admin` 模型配置页
- 目标：让前端节点参数条按具体模型能力展示，并让后台只负责路由启停与连通性管理

## 1. 背景

当前工作区的生成节点存在三个明显问题：

1. 节点里的比例选择不会驱动当前节点外框比例变化。
2. 视频节点缺少生成时长参数入口。
3. 后台模型配置与前台节点参数条之间的联动还不够清晰，尤其是“哪些模型该显示、显示哪些参数、如何测试 API 连通性”。

同时，项目当前阶段并不需要让管理员自行配置每个模型的详细参数模板。模型适配仍然由开发完成，用户后续只需要提供目标模型的 API 文档，由开发补齐前端模板与请求映射。

## 2. 设计目标

这轮设计只解决模型接入与参数展示的最小闭环：

1. 图片、视频、全景类节点的比例选择变成节点实例级设置，只影响当前节点。
2. 视频节点新增“生成时长”参数，使用滑动轴或等价的滑动交互。
3. 前台节点参数条改为读取开发侧预置的模型模板，按模型支持能力动态展示参数。
4. 后台模型配置页支持 API 连通性测试。
5. 前台只显示后台已启用、且与当前服务类型匹配的模型。

## 3. 非目标

这轮不做以下内容：

1. 不让管理员在后台维护每个模型的完整参数模板。
2. 不实现视频播放时间轴或可拖拽播放进度条。
3. 不做通用的“任意厂商零代码接入”能力。
4. 不改动任务队列、空间切换、历史瀑布流等与本需求无关的模块。

## 4. 核心边界

模型接入职责分两层：

### 4.1 后台负责

后台模型配置页继续负责路由层配置：

1. 服务类型
2. 厂商
3. Base URL
4. API Key
5. submit/query endpoint
6. 启用/停用
7. 优先级 / 默认路由
8. API 连通性测试结果

### 4.2 前端 / 开发负责

前端代码内维护具体模型的参数模板与请求适配：

1. 某个模型支持哪些参数
2. 参数条顺序与展示文案
3. 模式、比例、分辨率、时长的可选项
4. 参数如何映射到请求体
5. 不同模型的响应结果如何回填到节点

该模板由开发维护，不在后台开放编辑。

## 5. 模型模板设计

前端新增一个“模型模板注册表”，按具体模型名挂载能力定义。

建议结构：

```ts
type ModelParamTemplate = {
  vendor: string;
  serviceType: "text" | "image" | "video" | "audio";
  modelName: string;
  supportsMode?: boolean;
  supportsResolution?: boolean;
  supportsAspectRatio?: boolean;
  supportsAutoAspect?: boolean;
  supportsDuration?: boolean;
  modeOptions?: string[];
  resolutionOptions?: string[];
  aspectRatioOptions?: string[];
  durationRange?: {
    min: number;
    max: number;
    step: number;
    defaultValue: number;
  };
  defaults?: {
    mode?: string;
    resolution?: string;
    aspectRatio?: string;
  };
};
```

这份注册表只由开发维护。后续新增模型时，流程是：

1. 用户提供 API 文档
2. 开发增加或修改对应模型模板
3. 开发实现该模型的请求映射
4. 后台启用该模型后，前台即可显示

## 6. 前端节点参数条

## 6.1 适用节点

参数条的动态模板能力优先作用于：

1. 图片节点
2. 视频节点
3. 全景节点

文本和音频节点可先只保留当前必要参数，不强行补齐整套视觉结构。

## 6.2 参数条顺序

生成节点底部参数条顺序固定为：

1. 厂商
2. 模型
3. 模型模式
4. 自适应 / 比例 / 分辨率
5. 时间

但每一项是否显示，取决于当前选中模型模板是否支持。

例如：

1. 模型不支持 `mode`，则第 3 项不显示。
2. 模型不支持 `duration`，则第 5 项不显示。
3. 模型只支持分辨率不支持比例，则第 4 项只显示分辨率入口。
4. 模型支持自动比例，则第 4 项显示“自适应”状态与比例面板。

## 6.3 比例与节点尺寸

比例选择必须是节点实例级状态，而不是类型全局状态。

行为规则：

1. 用户在某一个图片/视频/全景节点上选择 `5:4`
2. 只更新该节点自己的比例配置
3. 当前节点的外框和预览区域同步切换到对应比例
4. 其他同类型节点保持自己的原比例，不受影响

这条规则优先于任何全局默认逻辑。

## 6.4 视频时长

视频节点本轮不做播放时间轴，只做“生成时长参数”。

交互规则：

1. 如果当前模型模板支持时长，参数条末尾显示时长入口
2. 时长入口展开后使用滑动轴或等价的滑动面板
3. 滑动范围、步进和默认值由模型模板决定
4. 当前值以 `5s / 10s / 15s` 这类形式展示在参数条里

## 7. 后台模型配置页

## 7.1 前台显示规则

前台模型下拉必须严格遵守两条规则：

1. 只显示后台已启用的模型配置
2. 只显示与当前节点服务类型一致的模型

例如：

1. 图片节点只显示已启用的图片模型
2. 视频节点只显示已启用的视频模型
3. 文本节点只显示已启用的文本模型

如果某个旧节点已经绑定了一个后来被停用的模型：

1. 旧节点可以继续显示当前值
2. UI 需标记该模型已停用
3. 新节点与新选择不再展示该模型

## 7.2 API 连通性测试

后台模型配置页增加“测试连接”能力。

目标：

1. 帮管理员快速验证 Base URL、endpoint、API Key 是否可用
2. 避免前台节点选择了不可用路由后才暴露错误

预期结果：

1. 成功：显示“连接成功”或“认证通过”
2. 失败：显示简短错误原因，例如：
   - 认证失败
   - endpoint 不存在
   - 请求超时
   - 返回非 JSON

实现上可以优先使用一个轻量探测请求，不要求真正发起完整生成任务。

## 8. 数据流

整体数据流如下：

1. 后台配置模型路由并启用
2. 前台读取已启用配置，按服务类型过滤候选模型
3. 前端根据当前选中模型名，从代码内模板注册表获取参数定义
4. 参数条按模板动态渲染
5. 节点保存当前实例自己的参数值
6. 提交生成时，将节点参数映射为对应模型的请求体

## 9. UI 与实现建议

1. 厂商选择与模型选择拆开，避免一个下拉承担两层含义。
2. 模式选择使用普通下拉。
3. 分辨率与比例继续保留现有浮层风格，但要按模板裁剪显示内容。
4. 视频时长使用独立小面板，风格与当前浮层保持一致。
5. 当前节点外框比例变化应有即时反馈，不需要重新生成后才体现。

## 10. 测试要求

至少覆盖以下验证：

1. 启用 / 停用模型后，前台对应服务类型下拉是否同步变化
2. 某个节点修改比例后，是否只影响当前节点
3. 视频模型切换后，时长滑动轴是否只在支持时出现
4. 不同模型切换后，模式 / 分辨率 / 比例入口是否按模板显隐
5. 后台 API 测试成功与失败路径是否都有可读反馈
6. 构建与现有 store / model-config 测试不回归

## 11. 分阶段实施建议

建议分两步做：

### Phase A

1. 前端模型模板注册表
2. 节点实例级比例
3. 视频时长滑动轴
4. 参数条按模板显隐

### Phase B

1. 后台 API 连通性测试
2. 停用模型后的前台过滤与旧节点兼容提示
3. 模型请求映射与后续 API 文档适配

这样可以先把工作区交互收口，再补后台的验证链路。
